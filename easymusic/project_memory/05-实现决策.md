## 2026-04-27 决策 005

### 决策内容

把“生成完成”和“正式发布”拆成两个层次，并统一公共可见性口径。

### 结论

- `music_status` 继续表示生成状态
- `publish_status` 单独表示发布状态
- 我的作品允许作者查看全部状态
- 我点赞的作品、首页、最新、推荐、用户主页只展示 `publish_status = PUBLISHED`
- 作品详情页中，未发布作品仅作者可见
- 未发布作品不可点赞，不累计公共播放量

### 原因

- 生成完成不等于适合公开展示
- 后续 AI 封面、标签编辑、发布设置都需要草稿态承接
- 统一口径后，前台展示和后台统计更稳定

## 2026-04-27 决策 006

### 决策内容

点赞统计采用“行为明细表 + 聚合字段”双轨方案。

### 结论

- `music_info_action` 作为点赞行为明细
- `music_info.good_count` 作为展示和统计所用的汇总字段
- 点赞/取消点赞时实时同步 `good_count`
- 通过 `stage1_upgrade.sql` 对历史数据做一次回填

### 原因

- 列表页、详情页、后台管理都频繁读取点赞数
- 实时 count 明细表成本更高，也不利于后续统计看板
- 兼顾一致性与查询性能

## 2026-04-27 决策 007

### 决策内容

发布状态变更不再复用通用“仅更新非空字段”的 SQL。

### 结论

- 单独增加 `updateMusicPublishInfo`
- 显式更新 `publish_status` 和 `publish_time`
- 切回草稿时允许把 `publish_time` 置空

### 原因

- 通用动态 SQL 无法正确处理“字段需要更新为 null”的场景
- 发布状态属于生命周期字段，单独收口更安全
