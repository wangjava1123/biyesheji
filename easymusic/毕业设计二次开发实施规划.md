# EasyMusic 毕业设计二次开发实施规划

## 1. 目标定义

本次改造的总目标不是在现有源码上做局部换皮，而是把项目升级为一套更完整的 AIGC 音乐创作平台，形成清晰、可答辩、可展示的个人二次开发成果。

最终主线统一为：

**用户一句话描述需求 -> AI 扩写成结构化提示词 -> 生成音乐 -> AI 生成封面 -> 积分结算 -> 作品发布与统计**

在实现上述主线的同时，需要对当前项目中明显不合理、且改造成本较低的问题进行顺手优化，尤其是：

- 现有业务链路中的功能缺口
- 现有实现中的统计和交互问题
- 当前界面的信息层级、合理性和观感问题
- 移动端和桌面端布局的稳定性问题


## 2. 改造范围

### 2.1 必做功能

1. 智能提示词增强
2. 基于增强提示词的音乐生成
3. AI 封面生成
4. 提示词增强、音乐生成、封面生成的积分结算
5. 作品发布机制
6. 创作与平台统计能力
7. 前台核心页面的结构和视觉优化
8. 当前项目中低成本但价值高的不合理点修复

### 2.2 优先做低成本优化，不做大规模重写

本次优化遵循以下原则：

- 能复用现有模块的，尽量复用
- 能通过加字段和加服务实现的，不轻易整库重构
- 能修复现有 bug 的，优先修复
- 能明显提升展示效果的页面优先改
- 成本过高且对答辩价值有限的深度重写，暂不作为第一优先级


## 3. 当前项目现状判断

### 3.1 现有技术结构

项目是标准前后端分离结构：

- 前台：`easymusic-front/easymusic-front-web`
- 后台：`easymusic-front/easymusic-front-admin`
- 后端接口：`easymusic-java/easymusic-web`
- 后端管理接口：`easymusic-java/easymusic-admin`
- 公共业务层：`easymusic-java/easymusic-common`

当前已经具备以下基础能力：

- 用户注册、登录、积分
- 音乐生成任务创建
- 生成结果轮询/回调
- 音乐列表展示
- 封面手动上传
- 商品充值和支付
- 后台基础管理

### 3.2 现有主链路

当前链路主要是：

**用户自己填写 prompt -> 提交音乐生成 -> 扣积分 -> 生成结果回写 -> 前台展示 -> 手动上传封面**

这条链路能跑通，但还不够像完整的“智能创作平台”，主要问题在于：

- 用户必须自己写 prompt，门槛较高
- 音乐生成和封面生成未形成一体化流程
- 作品生成后缺少规范的发布环节
- 统计维度不完整，后台缺少看板
- 页面层次和交互逻辑偏演示项目风格


## 4. 当前项目中需要顺手优化的问题

以下问题优先纳入改造计划，因为修改成本相对可控，但价值很高。

### 4.1 点赞数统计存在明显缺口

当前点赞接口只维护了 `music_info_action` 记录，`music_info.good_count` 没有同步更新，导致：

- 列表展示的点赞数不可靠
- 作品详情和后台统计不稳定
- 创作者主页的获赞展示容易失真

本项属于明确 bug，必须顺手修复。

### 4.2 作品“生成完成”和“正式发布”没有分层

当前音乐生成完成后，基本就直接进入内容流展示，缺少：

- 草稿态
- 待完善态
- 已发布态
- 隐藏/下架态

这会导致作品生命周期设计过于粗糙，不利于后续做封面生成、标题优化、发布设置和统计。

### 4.3 创作页的信息结构不合理

当前创作页偏向“表单提交页”，问题包括：

- 用户输入门槛高
- 缺少自然语言意图入口
- 缺少提示词确认区
- 缺少音乐提示词和图片提示词的分离展示
- 缺少生成前的成本提示

本页是整套系统的核心，必须重构交互结构，但尽量基于现有页面扩展，不推翻整个工程。

### 4.4 封面能力过弱

当前只支持手动裁剪上传封面，缺少：

- AI 生成封面
- 封面生成历史
- 封面重生成
- 封面来源区分

这与“AIGC 音乐平台”的主题不匹配。

### 4.5 前台布局大量依赖固定高度和固定定位

当前多个页面大量使用 `100vh`、固定定位和写死高度，移动端和小屏下容易出现：

- 内容区域挤压
- 播放器遮挡
- 滚动区域不自然
- 信息密度不均衡

本次应做结构性优化，至少保证主要页面在桌面和移动端都稳定可用。

### 4.6 后台缺少数据看板

目前后台更像 CRUD 管理台，缺少：

- 创作次数统计
- 模型使用统计
- 积分消耗统计
- 封面生成统计
- 发布数据统计
- 热门作品和热门创作者统计

这部分非常适合毕业设计展示，应作为必做内容之一。


## 5. 目标系统方案

## 5.1 智能提示词增强

### 功能目标

用户只需要输入一句自然语言需求，系统自动输出结构化创作结果，至少包括：

- 音乐提示词
- 封面提示词
- 推荐标题
- 推荐风格
- 推荐情绪
- 推荐人声
- 推荐标签

### 典型流程

1. 用户输入一句话需求
2. 系统调用大模型进行提示词扩写
3. 返回结构化 JSON 结果
4. 前端展示并允许用户微调
5. 用户确认后再发起音乐生成

### 结构化输出建议

返回结果建议包含：

- `rawPrompt`
- `musicPrompt`
- `imagePrompt`
- `titleSuggestions`
- `musicGenre`
- `musicEmotion`
- `musicSex`
- `tags`
- `visualStyle`

### 实现方式

后端新增独立能力层：

- `PromptAssistController`
- `PromptAssistService`
- `PromptGenerateApi`

不要把提示词增强逻辑直接塞进现有音乐生成 Service，否则后续难以复用到封面生成。

### 数据设计

新增表：`prompt_optimize_record`

建议字段：

- `record_id`
- `user_id`
- `biz_type`
- `raw_prompt`
- `optimized_prompt`
- `structured_result`
- `music_id`
- `integral`
- `status`
- `create_time`

### 积分策略

建议：

- 基础扩写可以免费或低价
- 高级扩写可单独收取少量积分
- 若时间有限，可先做免费扩写，后续再接入扣分


## 5.2 基于增强提示词的音乐生成

### 功能目标

将“智能提示词增强”作为音乐生成的上游能力，用户不再必须自己写标准 prompt。

### 实现思路

- 保留现有音乐生成主链路
- 新增“原始一句话”和“优化后 prompt”的区分存储
- 音乐生成时优先使用 AI 优化后的 `musicPrompt`

### 数据改造建议

在 `music_creation` 上增加字段：

- `origin_prompt`
- `prompt_source_type`
- `prompt_record_id`

说明：

- `prompt` 保留为最终执行的 prompt
- `origin_prompt` 保存用户原始一句话
- `prompt_record_id` 关联提示词增强记录

### 兼容策略

为兼容旧逻辑：

- 老用户仍可直接手写 prompt
- 新用户可走一句话增强模式
- 两种模式共存


## 5.3 AI 封面生成

### 功能目标

音乐生成完成后，基于标题、歌词、风格、情绪和图片提示词生成封面图，自动写入作品封面。

### 典型流程

1. 用户在作品卡片或作品详情点击“AI 生成封面”
2. 系统读取当前作品信息和图片提示词
3. 扣除封面生成积分
4. 调用图片生成接口
5. 成功后上传图片到本地或 OSS
6. 回写 `music_info.cover`
7. 失败则退款并记录原因

### 实现方式

后端新增：

- `MusicCoverController`
- `MusicCoverService`
- `ImageCreateApi`

异步任务处理方式尽量复用当前音乐任务模式，不单独发明一套完全不同的任务管理机制。

### 数据设计

新增表：`music_cover_creation`

建议字段：

- `cover_id`
- `music_id`
- `user_id`
- `creation_id`
- `prompt`
- `style`
- `model`
- `integral`
- `task_id`
- `cover_url`
- `status`
- `fail_reason`
- `create_time`
- `finish_time`

### 对现有表的补充

在 `music_info` 增加：

- `cover_source`
- `cover_generate_count`

其中：

- `0` 手动上传
- `1` AI 生成


## 5.4 积分结算体系扩展

### 功能目标

将提示词增强、音乐生成、封面生成统一纳入现有积分体系。

### 需要新增的积分记录类型

在 `UserIntegralRecordTypeEnum` 中扩展：

- `PROMPT_OPTIMIZE`
- `PROMPT_OPTIMIZE_BACK`
- `CREATE_COVER`
- `CREATE_COVER_BACK`

### 结算规则建议

- 音乐生成：沿用现有扣分逻辑
- 封面生成：单独扣分
- 提示词增强：可先免费，后续再接积分
- 任务失败：自动退款

### 配置策略

建议通过 `sys_dict` 配置以下内容，而不是硬编码：

- 提示词增强价格
- 封面生成价格
- 不同模型价格
- 不同风格附加价


## 5.5 作品发布与统计

### 功能目标

让“生成完成”不等于“直接公开”，作品要具备可管理的发布过程。

### 发布机制建议

在 `music_info` 增加字段：

- `publish_status`
- `publish_time`
- `view_scope`
- `tags`
- `prompt_record_id`

状态建议：

- `0` 草稿
- `1` 已发布
- `2` 已隐藏

### 发布流程建议

1. 音乐生成完成后进入草稿态
2. 用户可修改标题、生成或上传封面、补充标签
3. 用户点击发布
4. 已发布作品进入首页、最新、用户主页等公共区域

### 统计范围

前台和后台都应具备统计能力：

- 用户维度
  - 创作次数
  - 发布次数
  - 封面生成次数
  - 总播放量
  - 总获赞量
- 平台维度
  - 总创作数
  - 成功率
  - 模型使用次数
  - 积分消耗
  - 热门标签
  - 热门作品
  - 热门创作者

### 统计实现建议

若时间紧，可以先做直接查询统计。

若希望后台展示更稳定，可新增一张汇总表：`music_stat_daily`，通过定时任务按天汇总：

- 创作数
- 成功数
- 封面生成数
- 发布数
- 充值积分
- 消耗积分

## 8. 技术实现方案

## 8.1 外部 AI 配置文件约定

后续“提示词增强”和“AI 封面生成”依赖的外部 AI 服务配置，当前统一来源于项目根目录下的文件：

`D:\ProjectAllOk\Easymusic\easymusic\easymusic\api的url和key.txt`

该文件用于保存后续接入 AI 服务所需的信息，至少包括：

- 提示词增强接口 URL
- 提示词增强接口 Key
- 图片生成接口 URL
- 图片生成接口 Key

实现要求：

- 业务代码中不要直接硬编码 URL 和 Key
- 建议在后端增加统一配置读取层，将该文件内容映射为配置对象
- `PromptGenerateApi` 和 `ImageCreateApi` 统一从配置层获取地址和鉴权信息
- 后续如果切换供应商，只调整配置读取和适配层，不直接改业务流程代码

安全要求：

- 文档中只记录文件路径和用途，不记录实际密钥内容
- 实际开发时不要在日志中打印完整 Key
- 若后续准备提交到公开仓库，应将该文件加入忽略列表，避免密钥泄露

## 10. 验收标准

满足以下条件可视为本次改造完成：

1. 用户可用一句话触发提示词增强
2. 系统能返回结构化音乐提示词和图片提示词
3. 用户可使用增强后的提示词生成音乐
4. 用户可对生成作品发起 AI 封面生成
5. 积分扣减、失败退款、记录落库可用
6. 作品具备草稿、发布、隐藏等状态
7. 前台页面结构明显优于原项目
8. 后台可以看到核心统计数据
9. 当前已发现的低成本不合理点已完成修正


## 11. 风险与控制

### 风险 1：第三方 AI 接口不稳定

处理方式：

- 接口抽象，不把供应商写死在业务代码里
- 保留失败重试和退款逻辑

### 风险 2：一次性改动过大

处理方式：

- 按阶段推进
- 每一阶段形成可运行版本

### 风险 3：界面重构影响现有功能

处理方式：

- 优先重构核心页面
- 保留旧接口兼容层

### 风险 4：统计口径混乱

处理方式：

- 先统一作品状态和点赞逻辑
- 再做后台统计


## 12. 本次开发的落地原则

如果开始正式编码，本次开发按以下原则执行：

1. 优先修复现有明显 bug 和低成本不合理点
2. 优先完成“提示词增强 -> 音乐生成 -> 封面生成 -> 发布”的闭环
3. 优先改造创作页、我的作品页、作品卡片、作品详情页
4. 优先保证桌面端和移动端主要路径可用
5. 最后再做统计看板和视觉收尾


## 13. 结论

本次改造的核心不是增加几个零散功能，而是把原项目升级为一套具备以下特征的系统：

- 支持自然语言创作意图输入
- 支持结构化提示词增强
- 支持音乐与封面联动生成
- 支持积分结算与失败补偿
- 支持作品发布管理
- 支持平台统计和后台展示

**基于 Spring Boot + Vue 的 AIGC 音乐生成与多模态内容发布平台**
